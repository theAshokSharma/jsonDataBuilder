Options File Reference Guide
How to create and configure options.json
Version .1

1. What is the Options File?

The options.json file is an optional companion to your schema file. While schema.json defines data structure and field types, the options file controls how fields are presented — what input control to use, what values are available, and how fields interact.

Without an options file every field renders as a plain text or number input. The options file upgrades these to dropdowns, checkboxes, radio buttons, sliders, and date pickers, and adds dependent dropdowns, conditional field hiding, and shared value lists.

FilePurposeRequired?
schema.jsonDefines fields, types, validation rulesYes
options.jsonControls UI — input types, choices, dependenciesNo (but recommended)
data.jsonThe actual data being edited or createdNo (new record)

2. Top-Level File Structure

An options.json is a JSON object. Top-level keys correspond to field paths in your schema, plus two special reserved keys:

{
  "_lists": { ... },            // Optional: shared value lists
  "conditional_rules": { ... }, // Optional: field-disable logic

  "fieldPath": {                // One entry per field to configure
    "input_control": "...",
    "response_type": "...",
    "values": [ ... ]
  },

  "section.fieldPath": { ... }  // Nested fields use dot notation
}
ℹ️ Naming: The key must exactly match the field path in schema.json. For nested fields use dot notation: "demographic.mbr_first_name". Typos silently ignore the config.

3. Field Configuration Properties

PropertyTypeRequiredDescription
input_controlstringYesThe UI widget to render
response_typestringDropdowns onlysingle-select or multi-select
valuesarrayUsually yesThe selectable option values
nastring / objectNoAn explicit "Not Applicable" option
exclusive_valuesarrayNoValues that clear all others when selected
disable_valuesarrayNoVisible but not user-selectable
dependent_valuesobjectNoOptions that change based on another field

4. Input Control Types

4.1 drop-down

Renders a dropdown list. Use response_type to choose single or multi-value selection.

"marital_status": {
  "input_control": "drop-down",
  "response_type": "single-select",
  "values": [
    { "value": "S", "label": "Single" },
    { "value": "M", "label": "Married" },
    { "value": "D", "label": "Divorced" }
  ]
}

"allergies": {
  "input_control": "drop-down",
  "response_type": "multi-select",
  "values": [
    { "value": "PEANUT", "label": "Peanuts" },
    { "value": "DAIRY",  "label": "Dairy"   }
  ]
}

4.2 check-box

Scrollable checkbox list. Always stores an array regardless of response_type.

"symptoms": {
  "input_control": "check-box",
  "values": [
    { "value": "FEVER", "label": "Fever" },
    { "value": "COUGH", "label": "Cough" },
    { "value": "NONE",  "label": "None"  }
  ],
  "exclusive_values": ["NONE"]
}

4.3 radio-button

List of radio buttons. Always stores a single value.

"gender": {
  "input_control": "radio-button",
  "values": [
    { "value": "M", "label": "Male"   },
    { "value": "F", "label": "Female" },
    { "value": "O", "label": "Other"  }
  ]
}

4.4 date-time-picker

Native date / datetime-local / time picker. Format is derived from the schema field's format property (date, date-time, or time). No values array needed.

"birth_date": { "input_control": "date-time-picker" }

"appt_time": {
  "input_control": "date-time-picker"
  // Schema must have "format": "date-time" or "format": "time"
}

4.5 slider

Range slider. The values array defines numeric endpoints; min, max, and step are calculated automatically.

"pain_scale": {
  "input_control": "slider",
  "values": ["0", "10"]    // min=0, max=10, step=1
}

5. Values Format

The values array supports three formats (mixable):

5.1 Value / Label Object (recommended)

Stores the code; displays the label. The saved data file contains codes, not labels.

"values": [
  { "value": "CK001", "label": "Yes"     },
  { "value": "CK002", "label": "No"      },
  { "value": "CK003", "label": "Unknown" }
]

5.2 Plain String

Same string used as both stored value and displayed label.

"values": ["Active", "Inactive", "Pending"]

5.3 Numeric Range Shorthand

Expands "start-end" into consecutive integers — useful for age, year, or score fields.

"values": ["1900-2024"]   // expands to 1900, 1901, ... 2024
"values": ["0-100"]       // 101 options
⚠️ Consistency: exclusive_values, disable_values, and dependent_values keys must use the same value codes as values — not the label strings.

6. Special Option Features

6.1 exclusive_values

Selecting an exclusive value clears all other selections. Typical use: "None of the above", "N/A", "Unknown".

"chronic_conditions": {
  "input_control": "check-box",
  "values": [
    { "value": "DIA", "label": "Diabetes"    },
    { "value": "HTN", "label": "Hypertension"},
    { "value": "NA",  "label": "None"        },
    { "value": "UNK", "label": "Unknown"     }
  ],
  "exclusive_values": ["NA", "UNK"]
}

6.2 disable_values

Always shown at the bottom of the list (with a visual separator) but not user-selectable. Used for legacy or system-set values that may appear in loaded data — once cleared they cannot be re-selected.

"tobacco_status": {
  "input_control": "drop-down",
  "response_type": "single-select",
  "values": [
    { "value": "SMOKER", "label": "Current Smoker" },
    { "value": "NEVER",  "label": "Never Smoked"   }
  ],
  "disable_values": [
    { "value": "LEGACY", "label": "Legacy Record (not re-selectable)" }
  ]
}
ℹ️ If loaded data contains a disable_values entry the app displays it as selected. Once the user changes the field it cannot be re-selected.

6.3 na — Explicit Not-Applicable

Adds a dedicated N/A option stored separately from normal values. Behaves as exclusive in multi-select.

"secondary_diagnosis": {
  "input_control": "drop-down",
  "response_type": "multi-select",
  "values": [ { "value": "D01", "label": "Condition A" } ],
  "na": { "value": "NA", "label": "Not Applicable" }
}

7. Dependent Values

A field's options change dynamically based on another field's current value.

7.1 Basic Dependent Dropdown

"country": {
  "input_control": "drop-down",
  "response_type": "single-select",
  "values": [
    { "value": "US", "label": "United States" },
    { "value": "CA", "label": "Canada"        }
  ]
},
"state_province": {
  "input_control": "drop-down",
  "response_type": "single-select",
  "values": [],                    // empty by default; filled by trigger
  "dependent_values": {
    "country": {                   // trigger field name
      "US": [
        { "value": "CA", "label": "California" },
        { "value": "TX", "label": "Texas"      },
        { "value": "NY", "label": "New York"   }
      ],
      "CA": [
        { "value": "ON", "label": "Ontario"          },
        { "value": "BC", "label": "British Columbia" }
      ]
    }
  }
}

7.2 Dependent Values with Input Control Override

Each trigger value can supply its own input_control and response_type, changing the dependent field's widget type.

"diagnosis_code": {
  "input_control": "drop-down",
  "response_type": "single-select",
  "values": [],
  "dependent_values": {
    "diagnosis_type": {
      "SINGLE": {
        "input_control": "radio-button",
        "values": [ { "value": "D01", "label": "Code A" } ]
      },
      "MULTI": {
        "input_control": "check-box",
        "values": [
          { "value": "D01", "label": "Code A" },
          { "value": "D02", "label": "Code B" }
        ]
      }
    }
  }
}
ℹ️ When the trigger changes, the dependent field resets and options refresh. When loading data, the app pre-initializes dependent options before populating values.

8. Conditional Rules

Automatically disables (greys out) fields when specific conditions are met. Disabled fields are cleared and cannot be interacted with.

"conditional_rules": {
  "triggerField": [
    {
      "value": "TRIGGER_VALUE",
      "disable_fields": ["field_a", "section.field_b"]
    }
  ]
}

8.1 Example — Smoking History

"conditional_rules": {
  "tobacco_status": [
    {
      "value": "NEVER",
      "disable_fields": ["cig_per_day", "smoking_start_year", "pack_years"]
    }
  ]
}

8.2 Multiple Conditions on the Same Trigger

"conditional_rules": {
  "employment_status": [
    { "value": "RETIRED",    "disable_fields": ["employer_name", "work_hours"] },
    { "value": "UNEMPLOYED", "disable_fields": ["employer_name", "work_hours", "industry"] }
  ]
}
⚠️ A field is disabled if any matching condition targets it, and re-enabled only if no active condition targets it. Conditions can safely overlap.

9. Shared Value Lists (_lists)

Define repeated value sets once and reference them with $ref or the ## shorthand.

{
  "_lists": {
    "yesNoUnknown": [
      { "value": "Y",   "label": "Yes"     },
      { "value": "N",   "label": "No"      },
      { "value": "UNK", "label": "Unknown" }
    ]
  },
  "has_diabetes":    { "input_control": "radio-button", "values": { "$ref": "#/_lists/yesNoUnknown" } },
  "has_hypertension":{ "input_control": "radio-button", "values": { "$ref": "#/_lists/yesNoUnknown" } }
}

// In dependent_values use the ## shorthand:
"state": {
  "dependent_values": {
    "country": { "US": "##usStates" }
  }
}

10. Naming Convention

The app auto-detects the options file by replacing schema with options in the filename (case-insensitive):

Schema FilenameExpected Options Filename

insurance-schema.jsoninsurance-options.json
claim_schema.jsonclaim_options.json
my-data-schema.jsonmy-data-options.json

13. Validation & Troubleshooting

When an options file loads, the app validates all field keys against the schema. Missing keys trigger a scrollable warning dialog.

SymptomLikely Cause & Fix
Field still shows as plain textKey mismatch — check exact dot-notation path. Keys are case-sensitive.
Dependent field does not updateThe trigger key in dependent_values must match the schema field path exactly.
Value not found when loading dataData contains a value not in values[]. Field highlights red. Add it to values[] or disable_values[].
exclusive_values not workingEntries must match value codes, not label strings.
Validation warning: key not in schemaField path does not exist in schema. Check spelling, nesting, and case.
Conditional rule not disabling fieldThe rule's value must match the stored code, not the label.
